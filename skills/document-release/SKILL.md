# Document Release Skill

## Purpose

Update docs that became stale because of a change or release.

## When to Use

Use after implementation, before release, or during release preparation.

## Inputs Expected

- Change summary
- Changed files
- User-facing behavior changes
- Setup or deployment changes

## Process

1. Check existing docs first.
2. Update stale docs only.
3. Avoid duplicate docs.
4. Update templates if workflow changed.
5. Note docs intentionally left unchanged.

## Required Checks

- README drift
- Architecture docs
- Setup docs
- Changelog/progress notes
- Template updates
- User-facing behavior changes

## Output Format

- Docs checked
- Docs changed
- Why each doc changed
- Remaining docs gaps
- No-sprawl confirmation

## Stop Conditions

Stop if documentation scope expands beyond stale docs or requires product decisions.
